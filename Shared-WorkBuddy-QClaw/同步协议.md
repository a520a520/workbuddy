# WorkBuddy ↔ QClaw 双向同步协议

> 最后更新：2026-06-20
> 所有涉及腾讯文档的读写操作，必须遵守此协议

---

## 核心原则

**以「最后更新」时间戳为准，较新版本永远覆盖较旧版本。**

---

## 版本校验机制

### 时间戳格式

每份文档必须在开头或结尾包含：
```
> 最后更新：YYYY-MM-DD
```
或文末：
```
**最后更新：** YYYY-MM-DD
```

### 校验流程

```
步骤1：读取腾讯文档 → 提取 T_doc（最后更新时间）
步骤2：读取本地文件 → 提取 T_local（最后更新时间）
步骤3：比较 T_doc 与 T_local
步骤4：按同步规则执行
步骤5：同步完成后，更新双方时间戳为当日
```

### 同步规则

| 条件 | 操作 |
|------|------|
| `T_doc > T_local` | 腾讯文档较新 → 检查完整性 → 更新本地文件 |
| `T_local > T_doc` | 本地较新 → 检查完整性 → 更新腾讯文档 |
| `T_doc == T_local` | 时间相同 → 跳过，不同步 |
| 内容不完整 | 拒绝同步，输出警告信息 |

---

## 内容完整性检查标准

同步前必须检查内容是否完整，**不完整严禁同步**。

### 统一工作日志
- 必须包含段落：`本周重点` / `进行中任务`
- 行数 ≥ 30 行

### 新媒体素材制作计划
- 必须包含段落：`素材制作总览` / `短视频素材`
- 行数 ≥ 30 行

### 三少背景记忆（MEMORY.md）
- 必须包含段落：`硬性规则` / `业务版图` / `基本信息`
- 行数 ≥ 100 行

### QClaw项目记忆
- 必须包含段落：`QClaw的身份` / `共享数据源` / `你的工作方式`
- 行数 ≥ 50 行

**不完整的判定：** 缺少任意必含段落，或行数低于阈值 → 视为不完整，拒绝同步并报警

## 触发同步的时机

### WorkBuddy（主力写入方）
1. **用户说「工作到这里啦」/「我们回头见」等结束对话语** → 立即执行完整同步
2. **启动对话时** → 读取腾讯文档，检查是否有比本地更新的版本
3. **每次写入关键文件后** → 同步该文件到腾讯文档

### QClaw（追踪方，家里Mac Mini）
1. **每日9:00简报前** → 先拉取腾讯文档最新版本
2. **每2小时进度检查前** → 先拉取腾讯文档最新版本
3. **收到三少微信指令写入后** → 立即同步到腾讯文档

---

## 冲突处理

### 时间戳相同但内容不同
- 以**内容更完整的一方**为准（行数更多、必含段落更全）
- 若完整性相同 → 以**本地版本**为准（本地是主力写入方）

### 腾讯文档读取失败
- 不执行同步，输出警告
- 继续使用本地版本工作
- 下次触发同步时重试

---

## 实现注意事项

### 读取腾讯文档
- 工具：`mcp__tencent-docs__get_content`
- 读取后第一时间提取 `最后更新` 时间戳
- 同时进行完整性检查

### 写入腾讯文档
- 工具：`mcp__tencent-docs__doc.insert_markdown`（直接传 markdown，不用 base64）
- 写入前更新 `最后更新` 时间戳为当日
- 写入后验证：重新读取，确认内容正确

### 本地文件
- 写入前更新 `最后更新` 时间戳为当日
- 与腾讯文档保持相同的时间戳

---

## 报警信息格式

当同步被拒绝时，输出：

```
⚠️ 同步被拒绝
原因：[内容不完整 / 时间戳异常 / 读取失败]
文件：[文件名 / 文档ID]
建议：请手动检查文件内容
```
